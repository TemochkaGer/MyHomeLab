# Конспект
Традиционная виртуализация
-----------------------------
|          Host OS           |
-----------------------------
| Hypervisor(KVM,VMware,VBox)|
-----------------------------
|GuestOS |GuestOS  |GuestOS  |
-----------------------------
| App A  |  App B  |  App C  |
-----------------------------

## Проблемы VM:
 - Каждая ВМ требует полную ОС (2-4 Гб на образ)
 - Долгий запуск (30-60 секунд)
 - Высокое потребление ресурсов (RAM, CPU)
 - Гипервизор - дополнительный слой обстракции

 # Контейнеризация (Docker)

-------------------------------------
|              Host OS              |
-------------------------------------
| Docker Engine (containerd, runc)  |
-------------------------------------
|Container(proc A)|Container(proc B)|
-------------------------------------

## Ключевые отличия
 - Изоляция - Виртуальная машина - Контейнер
 - Изоляция - Полная (своя ОС) - Процессная(namespaces)
 - Размер - 2-4 Гб - 10-500 Мб
 - Запуск - 30-60 сек - 1-5 сек
 - Производительность - 5-15 % оверхед - 1-3 % оверхед
 - Ядро - Свое ядро операционной системы - Общее ядро хоста

## Как работает коетейнер
Контейнер - комбинация трех технологий Linux:
 1. Namespaces (Изоляция) - изолирует ресурсы системы для процесса:
    - Тип namespace - Что изолирует - Пример
    - PID - Процессы - Контейнер видит только свои процессы
    - NET - Сеть - Свой IP, порты, Интерфейсы
    - MNT - Файловая система - Свое дерево файлов
    - UTS - Имя хоста - Свой hostname
    - IPC - Межпроцессорное взаимодействие - Изолированная память
    - USER - Пользователи - Свой root внутри контейнера

Проверка namespaces на сервере: ls -la /proc/$$/ns/
 2. Control Groups (cgroups) (Ограничение ресурсов)
    - Ресурс - Ограничение - Зачем
    - CPU - 50% ядра - Чтобы контейнер не захватил все
    - Memory - 512 МБ - Защита от OOM (Out Of Memory)
    - I/O  - 100 Мб/с - Контроль дисковой нагрузки
    - Network - 1 Гбит/с - Контроль сети

Просмотр cgroups для контейнера: cat /sys/fs/cgroup/memory/docker/<container-id>/memory.limit_in_bytes

 3. Union File Systems (Слои Образа)
    Docker использует слоистую файловую систему (overlay2):
    -------------------------------
    |  Read-Write Container Layer  | <-- Изменения пользователя
    --------------------------------
    |  Read-Only Image Layer 3     | <-- Приложение
    --------------------------------
    |  Read-Only Image Layer 2     | <-- Зависимости
    --------------------------------
    |  Read-Only Image Layer 1     | <-- Базовая ОС
    --------------------------------
    |  Base Image                  | <-- Образ
    --------------------------------

Преимущества слоев:
 - Кэширование при сборке (изменился слой -> пересобираются только верхние)
 - Экономия места (общие слои не дублируются)
 - Быстрый деплой (скачиваются только измененные слои)
